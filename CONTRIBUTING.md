# Contributing to Awesome Radio Map Estimation

Thank you for helping grow this research hub. Contributions are welcome from
everyone — whether you spotted a missing paper, want to correct metadata, or
are adding a newly published result.

---

## How to add a paper

There are two ways:

### Option A — Open an Issue (easiest)

Click **New Issue** and choose **Add a paper**. The form guides you through
every field with dropdowns for the enum values, so no YAML knowledge is needed.
A maintainer will review the submission and create the data file.

### Option B — Submit a Pull Request

1. **One file per paper.** Create exactly one new YAML file under
   `data/papers/`. The filename must be the paper's arXiv id (e.g.
   `2210.12345.yml`) or a kebab-case slug if no arXiv id exists.
2. **Follow the schema.** See [`schema/README.md`](schema/README.md) for the
   complete field reference. All required fields must be present and every
   enum field must use one of the exact values listed there.
3. **Do not hand-edit `README.md`.** The paper list and statistics sections
   in the README are marked with `<!-- AUTO:* -->` comments and are
   **generated automatically** by CI from the YAML data. Edits to those
   regions will be overwritten on the next push to `main`. Only edit the
   hand-written sections above and below the markers if you have a genuine
   prose correction.
4. **One paper per PR.** Keeping each PR to a single file makes review fast
   and keeps the Git history clean.
5. **Verify locally (optional but appreciated):**
   ```bash
   pip install -r requirements.txt
   python scripts/validate.py
   ```
   A passing `validate.py` means CI will pass on the schema-validation job.

## The schema at a glance

Every file in `data/papers/` is a single YAML object validated against
[`schema/paper.schema.json`](schema/paper.schema.json). The human-readable
field reference lives in [`schema/README.md`](schema/README.md).

**Required fields:** `id`, `title`, `authors`, `year`, `subfield`,
`method_family`, `task`, `input_modality`, `environment`, `dimensionality`,
`supervision`, `added_date`.

**Key rules:**
- `id` must equal the filename stem (e.g. file `2210.12345.yml` → `id: "2210.12345"`).
- Enum fields must use exact values — see `schema/README.md` for the full lists.
- `task` and `method_detail` are **arrays** (a paper can belong to more than
  one task or use more than one method technique).
- Unknown or misspelled keys fail CI immediately (`additionalProperties: false`).

---

## CI checks

Two automated checks run on every PR that touches `data/` or `schema/`:

| Check | What it does |
|-------|-------------|
| **validate** (`validate.yml`) | Runs `python scripts/validate.py` — schema validation for all YAML files plus cross-file integrity (id = filename, no duplicate ids, all dataset references resolve). |
| **deploy** (`deploy.yml`) | Runs on merge to `main` only — regenerates `README.md` and `site/src/data/stats.json`, rebuilds the Astro site, and publishes to GitHub Pages. |

A PR that fails `validate.yml` will not be merged until the data file is fixed.

---

## Notes for maintainers

- The README's `<!-- AUTO:* -->` regions are rewritten by `scripts/build_readme.py`
  on every push to `main`. Never hand-edit those regions.
- `site/src/data/stats.json` is likewise generated by `scripts/stats.py`. Do
  not commit hand-edited versions of that file.
- `site/src/lib/types.ts` is generated from `schema/paper.schema.json` at
  build time by `site/scripts/gen-types.mjs`. Do not hand-edit it either.
